Аутентификация и безопасность
=============================

Вход и выход
------------

Вход выполняется по email и паролю. После успешного входа создаётся сессия (хранится в PostgreSQL). Выход очищает сессию.

Смена пароля при первом входе
-----------------------------

Если для пользователя установлен флаг «обязательная смена пароля», после входа отображается экран смены пароля. Продолжить работу можно только после установки нового пароля.

Двухфакторная аутентификация (2FA)
----------------------------------

Поддерживается TOTP (приложения типа Google Authenticator, Authy). После включения 2FA при каждом входе после ввода пароля нужно ввести код из приложения.

**Настройка 2FA (первый раз):**

#. В профиле пользователя выбрать включение двухфакторной аутентификации.
#. Отсканировать QR-код в приложении-аутентификаторе или ввести секрет вручную.
#. Ввести сгенерированный код для подтверждения.

.. image:: images/placeholder.png
   :alt: Настройка 2FA — QR-код и поле кода
   :align: center
   :width: 70%

**Проверка кода при входе:** после ввода пароля отображается форма ввода кода 2FA. После успешной проверки сессия считается «прошедшей 2FA» и доступ к приложению разрешён.

Восстановление пароля
---------------------

На экране «Забыли пароль?» доступны два варианта:

* **По ссылке из письма:** кнопка отправки ссылки на email (если на сервере настроен SMTP — в переменных окружения или в админ-разделе «Почта»). Переход по ссылке открывает страницу сброса (например, ``/reset-password?token=...``). Подходит, когда **2FA выключена** (сброс только по коду 2FA в этом случае недоступен).
* **По коду 2FA:** если у пользователя включён TOTP — новый пароль задаётся по коду из приложения-аутентификатора (без почты).

Настройки почты для отправки писем сброса задаются в админ-кабинете (раздел «Почта»). Тест подключения доступен там же.

Профиль пользователя
--------------------

В профиле (модальное окно из шапки) можно изменить имя, доску по умолчанию, аватар (пресет или загрузка), включить/выключить 2FA и уведомления по email о карточках.

Доверенные устройства (опционально)
-----------------------------------

В backend поддерживается модель «доверенных устройств» (TrustedDevice): при определённых условиях можно не требовать 2FA при повторном входе с того же устройства. Точное поведение (какие устройства считаются доверенными и на сколько) задаётся логикой приложения.

Перенос базы данных и 2FA
-------------------------

При переносе дампа TOTP-секреты остаются в таблице пользователей (они **не** привязаны к имени сервера). Старые сессии на новом сервере недействительны — нужно войти заново. Если коды из приложения не принимаются, чаще всего причина в **сильном сдвиге системного времени** на новой машине (настройте NTP). Если нужно срочно восстановить доступ без разбора причин, на целевом сервере в PostgreSQL отключите 2FA у всех пользователей (подставьте пользователя и имя БД из вашего ``docker/compose``):

.. code-block:: bash

   docker compose exec -T db psql -U kanban -d kanban -c \
     'UPDATE "User" SET "totpEnabled" = false, "totpSecret" = null, "totpTempSecret" = null;'

После этого все пользователи входят по email и паролю. При необходимости 2FA можно снова включить в профиле (новый QR-код и секрет).

Итог по разделам Auth API: профиль (GET/PATCH), логин/логаут, смена пароля, сброс по email и по TOTP, настройка и проверка 2FA, аватар, настройки почты (для администратора) — см. :doc:`api`.
